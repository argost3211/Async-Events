# Events — асинхронная система событий

Pet-проект высоконагруженного асинхронного взаимодействия между микросервисами на базе **Kafka** и **PostgreSQL**. Демонстрирует проектирование пропускной способности, работу с брокером сообщений и типичные приёмы оптимизации.

---

## Архитектура

Репозиторий объединяет несколько микросервисов с общей базой данных и брокером сообщений.

| Сервис | Назначение |
|--------|------------|
| **Event Generator** | Генерирует события с заданной интенсивностью, имитирует нагрузку на Producer (Semaphore + timeout), ретраи при недоступности Producer |
| **Producer** | Принимает события по API → пишет в Postgres → публикует в Kafka |
| **Consumer** | Читает события из Kafka → идемпотентно создаёт уведомления в БД |
| **DB** (Postgres + PgBouncer) | Хранение событий и уведомлений; лимит соединений и пулы — см. секцию БД |
| **Broker** (Kafka) | Доставка сообщений между Producer и Consumer; топик событий — 3 партиции, DLQ — 1 партиция |

## Ограничения и метрики

### Ресурсы микросервисов

| Сервис | Реплики | CPU | Пул соединений к БД |
|--------|---------|-----|---------------------|
| Producer | 1 | 1 | 30 (+30) |
| Event Generator | 1 | 1 | — |
| Consumer | 1 | 1 | 10 (+20) |

### БД (PostgreSQL)

| Параметр | Значение |
|----------|----------|
| Postgres `max_connections` | 100 |
| PgBouncer pool (transaction) | 90 соединений |


### Kafka (партиции)

| Топик | Партиций |
|-------|----------|
| События заказов (`order-events`) | 3 |

### Пропускная способность

| Метрика | Значение |
|---------|----------|
| RPS Producer | до **300** запросов/с |
| Response Time Producer p50 / p99 | 300 мс / 500 мс |
| Обработка событий в Consumer | до **100** сообщений/с |

### Покрытие тестами - 90 %

## Решённые задачи

- **Rate limit** на Event Generator — ограничение интенсивности для стабильной нагрузки на Producer.
- **BackgroundTask** для публикации в Kafka — быстрый ответ API без ожидания записи в брокер.
- **PgBouncer** — сокращение числа соединений к Postgres и упрощение управления пулом.
- **Сессия в обработчике** вместо `Depends` — быстрее возврат соединения в пул.

## Возможные улучшения
- Масштабировать базу данных (увеличить пул соединений).
- Увеличить число воркеров Producer и добавить балансировщик нагрузки.
- Масштабировать Consumer (больше реплик).
