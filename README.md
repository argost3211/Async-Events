# Events — высоконагруженная система обработки событий заказов

Pet-проект: асинхронная цепочка приёма событий заказов, записи в БД, публикации в Kafka и потребления с созданием уведомлений. Цель — продемонстрировать проектирование и достижение высокой пропускной способности при работе с брокером сообщений.

---

## Ключевые достижения

| Метрика | Результат |
|--------|-----------|
| Пропускная способность | до **X** событий/с (при конфигурации: N consumer’ов, M партиций) |
| Задержка (генератор → уведомление) | p99 **Y** мс (при необходимости — заполнить) |
| Масштабирование | горизонтальное масштабирование consumer’ов в одной consumer group |

*Конкретные цифры — после нагрузочных тестов; обновляются по мере измерений.*

---

## Архитектура

```
                    ┌──────────────────┐
                    │ event-generator  │  нагрузка (events/s)
                    │   :8002/metrics  │
                    └────────┬─────────┘
                             │ HTTP POST /api/v1/events
                             ▼
┌─────────────┐     ┌──────────────────┐     ┌─────────────┐
│   Postgres  │◀────│    producer      │────▶│   Kafka     │
│     :5432   │     │   :8000/metrics  │     │ order-events│
└─────────────┘     └──────────────────┘     └──────┬──────┘
                                                    │
                             ┌──────────────────────┘
                             ▼
                    ┌──────────────────┐     ┌─────────────┐
                    │    consumer(s)    │────▶│   Postgres  │
                    │   :8001/metrics   │     │notifications│
                    └──────────────────┘     └─────────────┘
```

- **event-generator** — генерирует события с заданной интенсивностью, отправляет на producer по HTTP.
- **producer** — принимает события (FastAPI), пишет в Postgres, публикует в Kafka; при сбое Kafka — фоновый job повторной публикации.
- **consumer** — читает из Kafka, идемпотентно создаёт уведомления в БД; retry и DLQ при ошибках.
- **Prometheus** — сбор метрик с generator, producer, consumer.
- **Grafana** — дашборд: throughput по цепочке, целевая vs фактическая скорость генератора, задержка ответа producer.

---

## Техники и решения

- **Асинхронный стек** — asyncio, asyncpg, aiokafka/FastStream; неблокирующий I/O по всей цепочке.
- **Партиционирование в Kafka** — ключ `user_id + order_id`: все события одного заказа в одной партиции, порядок сохранён.
- **Идемпотентность consumer’а** — уникальность по `(order_id, event_type)`; дубликаты не создают второе уведомление.
- **Отказоустойчивость** — retry с экспоненциальным backoff, при исчерпании попыток — отправка в DLQ и коммит offset.
- **Гарантия доставки** — событие сначала в БД, затем в Kafka; при падении Kafka — фоновый republish по непропубликованным записям.
- **Метрики и наблюдаемость** — Prometheus + Grafana: события/с по каждому звену, целевая vs фактическая скорость генератора, время ответа producer.

## Стек

- **Python** 3.11+
- **API:** FastAPI, Uvicorn
- **БД:** PostgreSQL, SQLAlchemy (async), asyncpg, Alembic
- **Брокер:** Apache Kafka (aiokafka / FastStream)
- **Мониторинг:** Prometheus, Grafana
- **Оркестрация:** Docker, Docker Compose

---

## Запуск

1. Скопировать `.env.example` в `.env` и при необходимости задать переменные (Postgres, Kafka).
2. Запустить стек:

   ```bash
   docker compose up -d
   ```

3. После старта:
   - **Producer API:** http://localhost:8000  
   - **Consumer metrics:** http://localhost:8001/metrics  
   - **Grafana:** http://localhost:3000 (admin / admin)  
   - **Prometheus:** http://localhost:9090  

Интенсивность генератора задаётся переменной окружения `GENERATOR_EVENTS_PER_SECOND` (по умолчанию — см. конфиг event-generator).
