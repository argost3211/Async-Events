## Purpose
Главные цели проекта:
- **Высокая пропускная способность** — реализовать и измерить большую пропускную способность системы.
- **Портфолио** — pet-проект, который станет преимуществом перед другими кандидатами при устройстве на работу; показать умение создавать высоконагруженную архитектуру и работать с брокерами сообщений (Kafka).

Система демонстрирует работу с Kafka и анализ пропускной способности. Стоит рассматривать consumer, producer и event_generator как отдельные сервисы.

Обязательные команды:

## Definition of Done
- покрытие тестами выше 85 процентов
- все тесты пройдены
- линтер black не имеет замечаний
- flake8 не имеет замечаний
- vulture не сообщает о мёртвом коде

## Canonical Documentation
- ARCHITECTURE.md
- logic.md
- REFERENCE_EXAMPLES.md

## MUST NOT (запрещено)
- создавать миграции без учета примеров миграций из @REFERENCE_EXAMPLES.md
- давать переменным имена без учета примеров из @REFERENCE_EXAMPLES.md
- запрещено обновлять метрики вне разрешенных мест
- реализовывать дополнительную логику, которая не описана в документации или запросе пользователя
- расширять сигнатуру методов:
    - ради тестов;
    - без согласования с пользователем;
    - без обоснования в виде бизнес-требования.
- `api/` **не содержит** логики работы с БД, Kafka или других многошаговых сценариев; только вызов use case и сервисов, маппинг в DTO, метрики, планирование фоновых задач.
- иметь неиспользуемые переменные или атрибуты класса или экземпляра в классах
- добавлять зависимости в файл @pyproject.toml или @poetry.lock без использования команды `poetry add <package>`

## MUST (обязательно)
- сначала нужно писать тесты, а потом реализацию (TDD)
- записывать изменени в файл @CHANGES.md в формате "Проблема" -> "Решение"
- следовать принципам SOLID
- реализовывать use case только как координатор вызовов; сложная логика и алгоритмы — в domain или в сервисах
- реализовывать сигнатуру методов строго в соответсвии с бизнес-требованиями, canonocal documentation или при согласовании с пользователем
- doc-string в классах и методах должны описывать только реализованный функционал
