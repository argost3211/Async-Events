# Логика приложения: события заказов и уведомления

Документ описывает архитектуру, потоки данных, контракты и практики pet-проекта: приём событий заказов по API, публикация в Kafka, потребление и формирование уведомлений с метриками и имитацией нагрузки.

---

## 1. Обзор архитектуры

### 1.1. Компоненты

| Компонент | Назначение |
|-----------|------------|
| **Producer** | HTTP API приём событий заказов → сохранение в Postgres → публикация в Kafka |
| **Consumer** | Чтение сообщений из Kafka → идемпотентная обработка → запись уведомлений в Postgres |
| **Broker (Kafka)** | Очередь событий между producer и consumer |
| **PostgreSQL** | Единая БД: сырые события заказов, уведомления, связь пользователь–заказ |
| **Event generator** | Отдельное приложение в Docker: непрерывная генерация событий с настраиваемой интенсивностью, задержками и дубликатами |

### 1.2. Поток данных (высокоуровнево)

```
[Event generator] --HTTP--> [Producer API] --> Postgres (события заказов)
                                    |
                                    v
                              [Kafka topic]
                                    |
                                    v
[Consumer] <-- consume --- [Kafka] --> Postgres (уведомления, связь user–order)
```

### 1.3. Инфраструктура

- Один экземпляр **PostgreSQL** в `docker-compose`; к нему подключаются и producer, и consumer.
- Одна **схема БД** для обоих сервисов.
- **Kafka** — один брокер; producer и consumer используют один и тот же брокер.
- **Масштабирование реплик**: **producer** — только **одна реплика** (один инстанс). Это нужно для фонового job повтора публикации в Kafka (п. 3.2.1) без координации между инстансами и чтобы не дублировать сообщения. **Consumer** может быть **несколько реплик**: инстансы работают в одной consumer group, Kafka распределяет партиции топика между ними; так повышается throughput обработки.
- Зависимости сервисов в `docker-compose.yml`: `db` и `broker` стартуют первыми; producer и consumer зависят от них; event generator зависит от producer.

## 2. События заказов (Order events)

### 2.1. Жизненный цикл заказа

Поддерживаются пять типов событий (полный цикл):

| Тип события | Описание |
|-------------|----------|
| `order_created` | Заказ создан |
| `order_paid` | Заказ оплачен |
| `order_shipped` | Заказ отгружен |
| `order_delivered` | Заказ доставлен |
| `order_cancelled` | Заказ отменён |

### 2.2. Payload события (API и Kafka)

В теле события должны быть:

- **order_id** — идентификатор заказа (UUID или строка), один и тот же для всей цепочки событий по заказу.
- **user_id** — идентификатор пользователя (связь пользователь–заказ; нужен для уведомлений).
- **event_type** — один из: `order_created`, `order_paid`, `order_shipped`, `order_delivered`, `order_cancelled`.
- **event_occurred_at** — время, когда событие произошло (приходит из event generator для каждого события).

Единая схема (Pydantic-модель) для API producer и для сообщений в Kafka, чтобы не было расхождений.

### 2.3. Валидация и порядок

- API producer валидирует `event_type` и обязательные поля.
- в логике обработки допускается, что generator может создавать и дубликаты — consumer должен быть к этому устойчив за счёт идемпотентности и DLQ.

## 3. Producer

### 3.1. Обязанности

1. Принять событие заказа по **HTTP API** (`POST /api/v1/events`).
2. **Валидировать** тело запроса через Pydantic-модель (event_type, order_id, user_id, event_occurred_at).
3. **Сохранить** событие в Postgres (обязательно каждое входящее событие).
4. **После успешной записи в БД** отправить сообщение в Kafka и обновить поле `published_to_kafka` в БД.
5. Вернуть клиенту успешный ответ (201 с id сохранённой записи).

### 3.2. Порядок операций

- Сначала **только** запись в БД (в транзакции).
- После **успешного** commit — публикация в Kafka.
- Если Kafka недоступна или публикация падает — событие уже есть в БД; применяется стратегия повтора публикации через фоновый job (см. п. 3.2.1).

#### 3.2.1. Фоновый job повтора публикации в Kafka

Периодическая задача должна запускаться вместе со стартом самого сервиса producer:

1. **Хранение состояния**: в таблице событий заказов есть поле `published_to_kafka` (boolean, по умолчанию `false`). После успешной отправки сообщения в Kafka producer выставляет `published_to_kafka = true` (отдельным запросом/транзакцией после publish).
2. **Выборка**: job с заданным интервалом (например, каждые 10–30 секунд) выбирает из БД записи, где `published_to_kafka = false` и `created_at` не старше заданного окна (например, последние 24 часа), с лимитом по количеству за один проход (batch).
3. **Публикация**: для каждой записи job формирует сообщение (как при обычной публикации), отправляет его в топик `order-events`. При успехе обновляет `published_to_kafka = true`.
4. **Повторы**: при ошибке отправки запись остаётся с `published_to_kafka = false` и будет обработана на следующем цикле.


### 3.3. Kafka (со стороны producer)

- **Топик**: один топик для всех событий заказов (`order-events`).
- **Ключ сообщения (key)**: составной из `user_id` + `order_id`, чтобы все события одного заказа и пользователя попадали в одну партицию и обрабатывались одним consumer в рамках партиции (порядок по заказу).
- **Значение (value)**: сериализованный payload события: JSON с `event_id`, `order_id`, `user_id`, `event_type`, `event_occurred_at`.

### 3.4. Политика создания топиков

Топики Kafka создаются **явно**, без опоры на auto-create:

- **Кто создаёт**: при старте **producer** проверяет существование топиков `order-events` и `order-events-dlq`; если топика нет — создаёт с заданными параметрами (число партиций, replication factor). Альтернатива: отдельный init-сервис или скрипт в docker-compose, который создаёт топики до запуска producer и consumer.
- **Параметры**: для `order-events` — число партиций согласовано с docker-compose (например, 3); для `order-events-dlq` — 1 партиция.
- **Consumer** не создаёт топики; предполагается, что к моменту подписки топик уже существует.

### 3.5. API (кратко)

- `POST /api/v1/events` — приём одного события заказа (body = payload из п. 2.2). Ответ: 201 + id записи в БД.
- `GET /api/v1/events` — список последних событий из БД для мониторинга и отладки.
- `GET /health` — Producer предоставляет **healthcheck**, используемый docker-compose и event-generator для проверки готовности API.

## 4. Consumer

### 4.1. Обязанности

1. Подписаться на топик `order-events`.
2. Читать сообщения, десериализовать payload.
3. **Идемпотентно** создавать уведомление для пользователя: по (order_id, event_type), не создавать дубликат.
4. Писать уведомления в **ту же БД**, в ту же схему (таблица уведомлений).
5. Поддерживать связь **пользователь–заказ** (user_id из события в поле в уведомлении).
6. Использовать **retry** при временных сбоях и **DLQ** для сообщений, которые после N попыток обработать не удалось.

### 4.2. Идемпотентность

- Ключ идемпотентности: пара `(order_id, event_type)`.
- Перед вставкой уведомления проверять: уже есть ли запись с таким ключом. Если есть — считать сообщение обработанным, не создавать второе уведомление, закоммитить offset (ack).
- **Commit offset** выполняется **только** после успешной записи уведомления в БД либо после проверки «уже обработано» (запись с таким (order_id, event_type) уже существует — тогда skip и commit). При повторной доставке одного и того же сообщения (например, из-за сбоя consumer до commit) повторная обработка по тому же (order_id, event_type) не создаёт второе уведомление благодаря этой проверке.

### 4.3. Retry

- При ошибке (БД недоступна, валидация и т.д.) не коммитить offset; повторить обработку сообщения через конфигурируемый интервал и ограниченное число попыток (например, 3–5).
- Использовать экспоненциальный backoff при повторах.
- После исчерпания попыток — отправить сообщение в **DLQ** (отдельный топик `order-events-dlq`) и закоммитить offset основного топика, чтобы не блокировать партицию.

### 4.4. Dead Letter Queue (DLQ)

- Топик `order-events-dlq`: сюда попадают сообщения, которые consumer не смог обработать после N попыток.
- В DLQ сохранять исходное тело сообщения + метаданные (причина ошибки, число попыток, timestamp). Дальнейшая обработка DLQ не требуется.

### 4.5. Уведомления в БД

- Таблица уведомлений: `id`, `user_id`, `order_id`, `event_type`, `message`, `created_at`.
- Связь пользователь–заказ: хранится через `user_id` и `order_id` в каждом уведомлении.
- **Поле message**: текст уведомления **генерируется по шаблону** в consumer в зависимости от `event_type` (например: «Заказ {order_id} создан», «Заказ {order_id} оплачен», «Заказ {order_id} отгружен» и т.д.). Шаблоны задаются в коде.

## 5. Общая база данных (схема)

- Один инстанс Postgres, **одна схема** для producer и consumer.
- Предполагаемые сущности (конкретные имена таблиц и полей — к реализации):
  - **События заказов (events)** (пишет producer): id, order_id, user_id, event_type, created_at (когда создана запись в БД), event_occurred_at (время наступления события, из API/генератора), published_to_kafka (boolean — опубликовано ли в Kafka; используется фоновым job, п. 3.2.1).
  - **Уведомления (notification)** (пишет consumer): id, user_id, order_id, event_type, message, created_at.

Миграции (Alembic) должны создавать/обновлять таблицы в этой общей схеме; producer и consumer используют одни и те же connection string из .env.

## 6. Метрики и эффективность архитектуры

Цель метрик — оценить, насколько система справляется с **высоким потоком событий** (throughput, задержки, потери, накопление лага).

### 6.1. Рекомендуемые метрики

| Метрика | Где считать | Назначение |
|---------|-------------|------------|
| **Число событий, принятых API (в единицу времени)** | Producer | Incoming rate (req/s). |
| **Число записей в БД (события заказов)** | Producer | Подтверждение, что каждое событие сохранено. |
| **Число сообщений, отправленных в Kafka** | Producer | Throughput публикации; сравнение с числом принятых запросов. |
| **Число сообщений, потреблённых из Kafka** | Consumer | Consume rate (msg/s). |
| **Число созданных уведомлений** | Consumer | Throughput обработки. |
| **Consumer lag** | Consumer / Kafka | Отставание consumer’а от последнего сообщения в партиции (критично при высокой нагрузке). |
| **Время обработки одного сообщения** | Consumer | Latency обработки (перцентили p50, p95, p99). |
| **Число retry** | Consumer | Частота временных сбоев. |
| **Число сообщений в DLQ** | Consumer / мониторинг DLQ | Количество «неудачных» сообщений. |
| **Ошибки записи в БД / ошибки Kafka** | Producer, Consumer | Частота сбоев по компонентам. |

### 6.2. Где хранить и отдавать метрики

- **Prometheus + Grafana**: producer и consumer отдают метрики в формате Prometheus; в docker-compose добавляются сервисы Prometheus и Grafana.
- **Эндпоинт `/metrics`**:
  - **Producer** и **Consumer** предоставляют HTTP-эндпоинт **`GET /metrics`**, который возвращает метрики в текстовом формате Prometheus (exposition format): строки вида `metric_name{label="value"} 123` или `metric_name 456`.
  - Эндпоинт доступен без аутентификации внутри сети docker; Prometheus периодически скрейпит `http://producer:8000/metrics` и `http://consumer:<port>/metrics` по своему `scrape_config`.
  - В Grafana добавляются дашборды с запросами к Prometheus (запросы к метрикам из п. 6.1): throughput, lag, latency, retry, DLQ и т.д.

### 6.3. Сценарии нагрузки

- Event generator настраивается на высокий rate (например, 100–1000 событий в секунду) и удерживается в течение нескольких минут.
- Наблюдаются: стабильность producer (нет 5xx, рост lag в допустимых пределах), рост consumer lag, время до стабилизации после пика, количество записей в DLQ и retry.

## 7. Event generator (имитация клиентов)

### 7.1. Назначение

Отдельное приложение в Docker, которое **постоянно** генерирует события заказов и отправляет их на API producer по HTTP. Нужно для демонстрации нагрузки, проверки идемпотентности (дубликаты) и реалистичности задержек.

### 7.2. Режим работы

- **Непрерывная генерация**: в цикле с настраиваемой интенсивностью (например, N запросов в секунду или интервал между запросами).
- **Логическая связность**: события по одному заказу должны образовывать осмысленную последовательность (например, created → paid → shipped → delivered или created → cancelled), но с **случайными задержками** между шагами и возможностью **повторов** (одно и то же событие отправляется дважды с небольшим интервалом) для проверки идемпотентности consumer’а.

### 7.3. Настраиваемые параметры (переменные окружения)

- **Базовый URL** producer API.
- **Интенсивность**: количество событий в единицу времени (например, `EVENTS_PER_SECOND=10`) или интервал между отправками.
- **Количество «виртуальных заказов»**: сколько заказов одновременно находятся в цикле (например, генерировать цепочки для 100 order_id по кругу).
- **Вероятность дубликата**: с какой вероятностью повторно отправить последнее событие того же заказа (для проверки идемпотентности).
- **Задержки**: мин/макс задержка (мс) между событиями одного заказа или между отправками запросов (случайная величина в заданном диапазоне).

### 7.4. Развёртывание

- Отдельный сервис в `docker-compose` (например, `event-generator`).
- Зависимость: от сервиса producer; **event-generator при запуске дожидается успешного healthcheck producer** — в docker-compose задаётся `depends_on: producer` с условием `service_healthy` (если используется Docker Compose с поддержкой condition), так что контейнер generator стартует только после того, как producer отчитается о готовности через healthcheck. Дополнительно при старте самого приложения generator опрашивает эндпоинт health producer (retry с backoff), и только после первого успешного ответа запускает цикл генерации событий.
- Параметры задаются через общий `.env` файл.

### 7.5. Поведение

- Генерация цепочек: для каждого «активного» order_id по таймеру/интервалу выбирается следующее событие в цепочке (created → paid → shipped → delivered или cancelled), с случайной задержкой и с заданной вероятностью дубликата предыдущего события.
- Отправка через HTTP асинхронно на `POST /api/v1/events`.
- Логирование: количество отправленных запросов, ошибок, при необходимости — отправленные order_id и event_type для отладки.

## 8. Docker Compose и зависимости

- **db** (Postgres): без зависимостей; должен быть первым.
- **broker** (Kafka): без зависимостей от других сервисов; поднимается вместе с db.
- **producer**: `depends_on: [db, broker]`; при старте должен обрабатывать недоступность Kafka (retry подключения) и БД (retry подключения и миграции).
- **consumer**: `depends_on: [db, broker]`; при старте — retry подключения к Kafka и БД, затем подписка на топик.
- **event-generator**: `depends_on: [producer]` с условием готовности producer: использовать **healthcheck** producer (в сервисе producer задаётся `healthcheck` с проверкой, например, `GET /health`); в `depends_on` для event-generator указать `producer` с условием `service_healthy`, чтобы generator не стартовал до успешного прохождения healthcheck у producer.

Конфигурация через env_file.

## 9. Краткое резюме потоков

1. **Event generator** непрерывно шлёт HTTP-запросы с событиями заказов (логичные цепочки + случайные задержки и дубликаты) на **Producer**.
2. **Producer** сохраняет каждое событие в **Postgres**, после успешной записи публикует в **Kafka**.
3. **Consumer** читает из Kafka, обрабатывает идемпотентно, при ошибках делает retry с backoff, после исчерпания попыток отправляет в **DLQ**, пишет **уведомления** в ту же **Postgres**.
4. **Метрики** собираются для оценки эффективности при большом потоке событий; реализация — экспорт в Prometheus и дашборд.
5. Всё разворачивается в Docker: одна БД, один брокер Kafka, producer, consumer, event-generator с зависимостями, описанными в п. 8.
